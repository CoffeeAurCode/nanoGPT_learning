## Concept Deep-Dives

### 1. Language Modeling: Next-Token Prediction

**Definition:** A language model assigns a probability to every possible next token given all previous tokens. Training means adjusting weights until the model assigns high probability to the actual next token.

**Minimal standalone example:**

```python
# Imagine we have a vocabulary of just 4 words: ["the", "cat", "sat", "mat"]
# Token IDs:                                       0       1      2      3

# The model sees: ["the", "cat"] → predicts probabilities for the next token
# Good model output: [0.05, 0.05, 0.85, 0.05]  (high prob for "sat")
# Training signal: cross-entropy loss against the true next token (ID=2)

import torch
import torch.nn.functional as F

logits = torch.tensor([[0.05, 0.05, 0.85, 0.05]])   # raw model output
target = torch.tensor([2])                            # "sat" is the true next token
loss = F.cross_entropy(logits, target)
print(f"loss: {loss.item():.4f}")   # lower = better prediction
```

**How nanoGPT uses it:** `train.py` calls `model(X, Y)` where `X` is a batch of token sequences and `Y` is the same sequences shifted one position to the right. The model produces logits and computes cross-entropy loss across all positions simultaneously.

**What breaks without it:** Without the shift (`Y = X[1:]`), the model would "predict" tokens it has already seen — trivially zero loss, zero learning.

### 2. The Autoregressive Property

**Definition:** "Autoregressive" means each new token is generated by feeding the model its own previous output. Generation is sequential; there is no parallelism at inference time.

**Minimal standalone example:**

```python
# Pseudo-code for autoregressive generation:
context = [stoi["ROMEO"]]          # start with one token
for step in range(max_new_tokens):
    probs = model(context)          # predict distribution over vocab
    next_token = sample(probs)      # pick a token
    context.append(next_token)      # feed it back in
    print(itos[next_token], end="")
```

**How nanoGPT uses it:** `GPT.generate()` in `model.py` implements exactly this loop, cropping the context to `block_size` if it grows too long.

**What breaks without it:** If you tried to generate all tokens in parallel (like BERT's masked language model), you would need to know all future tokens to generate each one — not possible for open-ended text generation.

---

### 3. The Transformer Decoder Architecture

**Definition:** A stack of identical blocks, each containing a causal self-attention layer followed by a position-wise MLP, with residual connections and layer normalization around each. "Causal" means token i can only attend to tokens 0…i, not the future.

**Visual map of nanoGPT's model:**

```
Input tokens (integers)
        │
    [Embedding: token + position]
        │
    [Transformer Block] × n_layer
    │   ├── LayerNorm
    │   ├── CausalSelfAttention
    │   ├── LayerNorm
    │   └── MLP
        │
    [Final LayerNorm]
        │
    [Linear: n_embd → vocab_size]   ← "lm_head"
        │
    Logits → cross-entropy loss (training) or softmax → sample (inference)
```

**How nanoGPT uses it:** `model.py` is the entire implementation. `GPTConfig` holds the hyperparameters. The `GPT` class builds the full stack.

**What breaks without it:** Without the causal mask in self-attention, the model could cheat by looking at future tokens during training — it would appear to converge but would generate garbage at inference time since those future tokens wouldn't exist.

## Reading the Source Files

### `config/train_shakespeare_char.py`

```python
out_dir = 'out-shakespeare-char'
eval_interval = 250
eval_iters = 200
log_interval = 10

always_save_checkpoint = False  # only save when val improves

dataset = 'shakespeare_char'
gradient_accumulation_steps = 1
batch_size = 64
block_size = 256   # context window: 256 characters

# baby GPT model
n_layer = 6
n_head = 6
n_embd = 384
dropout = 0.2

learning_rate = 1e-3
max_iters = 5000
lr_decay_iters = 5000
min_lr = 1e-4
beta2 = 0.99
warmup_iters = 100
```

This file is a plain Python config that will be `exec()`-ed by `configurator.py`. Every variable here overrides the corresponding default in `train.py`. Notice the model is tiny: 6 layers, 6 heads, 384-dim embeddings — about 10 million parameters. The full GPT-2 has 124 million.

### `data/shakespeare_char/prepare.py`

The prepare script does three things:
1. Downloads the 1 MB Tiny Shakespeare text file
2. Builds a character-level vocabulary (`stoi`: char→int, `itos`: int→char)
3. Encodes the text and writes `train.bin`, `val.bin`, and `meta.pkl`

The `meta.pkl` file is critical — it stores `stoi` and `itos` so `sample.py` can decode generated token IDs back to characters.

## Why This Design

**Why character-level tokenization for the starter config?**  
No external dependencies: no tiktoken, no BPE vocabulary. The vocab is just the 65 unique characters in Shakespeare. This makes the setup completely self-contained and runs in under 5 minutes on a laptop CPU. The trade-off: character-level models need longer sequences to learn the same patterns as BPE models, so they produce lower-quality text for the same model size.

**Why train a tiny model first?**  
Training GPT-2 (124M parameters) requires hours on a GPU and gigabytes of data. The `shakespeare_char` config produces visible, fun results in minutes. Same code, 12× smaller model.

## Exercises

**1 (Easy) — Count parameters:**  
Open a Python REPL after the training run completes and count the parameters in the model yourself:

```python
import torch
from model import GPTConfig, GPT
cfg = GPTConfig(n_layer=6, n_head=6, n_embd=384, block_size=256,
                vocab_size=65, dropout=0.0, bias=False)
model = GPT(cfg)
total = sum(p.numel() for p in model.parameters())
print(f"{total:,} parameters")
```

**Success condition:** You get a number close to what `train.py` printed when it started. Understand why position embeddings are subtracted from `get_num_params()`.

**2 (Medium) — Modify the model size:**  
Edit `config/train_shakespeare_char.py` to use `n_layer=2, n_head=2, n_embd=64`. Re-run training. Compare the final val loss and generated text quality to the 6-layer version. What do you observe?

**Success condition:** You can explain why the smaller model performs worse and estimate the parameter count ratio without running the code.

**3 (Hard) — Add a new config key:**  
Add a `max_seq_len_multiplier` key to `train_shakespeare_char.py` that sets `block_size = 128 * max_seq_len_multiplier`. Pass `--max_seq_len_multiplier=3` from the CLI and verify `block_size` becomes 384.

**Success condition:** The model trains with `block_size=384` when the multiplier is 3. Identify which two files you needed to change and why.
